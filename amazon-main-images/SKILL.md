# 亚马逊家居摆件主图全流程生产线

---

## 【第一铁律 · 全Skill最高优先级】

> Image_1 ~ Image_9 的英文描述中,产品本体描述必须极简。
> 此规则不可被任何其他规则覆盖,违反即强制重写。

```
- 只允许出现产品类别名(the sculpture / the vase / the figurine 等)
- 禁止任何颜色词、纹理词、造型形容词、工艺形容词
- 禁止任何尺寸词或厘米数字
- 细节/材质类图只能描述"镜头要拍什么属性",不能描述"产品长什么样"
- 尺寸图允许出现尺寸数字和测量线描述,因为这是图本身的功能

强制锁定:每张图描述的开头必须用
"The exact same [产品类别] from the reference image"
来锁定参考图,防止模型自由创作产品本体。
负面词必须包含 "no color change to the product"。

原因:Skill 默认图生图模式,产品本体由参考图承载。
描述产品外观=和参考图打架=生成产品与实物不一致。
```

---

## 【视觉哲学 · 第二最高原则】

> 与第一铁律并列,贯穿所有图片描述生成与审核。

```
Skill 生成的描述应该让模型像真实摄影师那样工作,
不是像修图师那样工作。

主角和配角的关系靠焦点、构图、光线、站位建立,
不靠把配角模糊或缩小来"伪装"主角的重要性。

道具按真实生活大小自然存在,产品按真实尺寸自然存在,
两者在同一画面里以真实的比例共处。

构图上产品始终是视觉中心,道具围绕产品但不越过产品的视觉领地。
```

这是方向,不是死规则。具体怎么落地由执行者凭对真实摄影的理解判断。
每张 Image_N 生成后必须经过视觉哲学审核(详见 G1)。

## 一、元原则(方向性)

1. 第一铁律高于一切
2. 视觉哲学贯穿全流程
3. Skill 默认图生图模式,所有 9 张图均基于参考图生成
4. 流程顺序:定位 → 数据解析 → 锁风格 → 生成风格包 → 锁心智方案 → 分配职责 → 逐图落地 → 校验
5. 风格包是 9 张图视觉一致性的唯一保证
6. 职责池只定"做什么",不定"用什么"
7. 9 张图默认场景+置景混搭

> 上述为方向性元原则,不可机械审核。
> 所有可量化、可机械审核的硬约束已迁移至 §二「全局硬约束清单」,该清单是全品类硬约束的**唯一声明源**。
> 品类专属硬约束在子 SKILL 内有独立清单(B2 §零 / B1 §零 等)。

## 二、全局硬约束清单(单点声明源)

> 本节是全品类硬约束的唯一声明源。
> 下游所有文件(M1/M2/D1/A2/G1/D2/S1/P1)只允许用 H 编号引用本节,不允许重复定义数字或修改阈值。
> 修改任何一条硬约束 → 只在本节修改一处即可全链路生效。

| 编号 | 内容 | 适用 | 审核位 | 违反后果 |
|---|---|---|---|---|
| **H1** | Image_2 必为尺寸图(硬锁,不可移位) | 全 | M2 Step 7 | 重新分配 |
| **H2** | 9 张图职责互不重复(同一种展示只做一张) | 全 | M2 Step 7 | 重新分配 |
| **H3** | 场景组职责数 ≥4 张 | 全 | M2 Step 7 | 重新分配 |
| **H4** | 场景全景数 ≥3 张(挂场景组下,置景全景不计) | 全 | M2 Step 7 | 重新分配 |
| **H5** | 交互组职责数 ≥2 张 | 全 | M2 Step 7 | 重新分配 |
| **H6** | 场景空间不同数 ≥3 个 | 全 | M2 Step 7 | 重新分配 |
| **H7** | 景别种类 ≥3 种 | 全 | M2 Step 7 | 重新分配 |
| **H8** | 机位组合(高度+方位+距离)≥3 种 | 全 | M2 Step 7 | 重新分配 |
| **H9** | 置景类落点统一(场景类按空间自然变化) | 全 | M2 Step 7 | 重新分配 |
| **H10** | 仅正面叙事面禁后角/纯侧面,所有图限正面±15°/微仰/微俯 | 全 | M2 Step 7 | 该图重写 |
| **H11** | 默认明亮安全区,暗调/戏剧化默认排除(题材强指向暗调除外) | 全 | M2 Step 4 | 风格包重生 |
| **H12** | 默认 1:1 比例 | 全 | M2 Step 8 | 该图重写 |
| **H13** | 非规格图必须显式声明镜头焦段 + 光圈 **硬锁 85mm + f/2.0**(类型 2 手部交互允许 50mm,Image_2 尺寸图豁免) | 全 | M2 Step 9 | 该图重写 |
| **H14** | 非规格图必须显式声明 `focus locked on the [产品类别]` + `strong foreground-background separation`(Image_2 豁免) | 全 | M2 Step 9 | 该图重写 |
| **H15** | 非规格图背景必须写成 `dissolving into soft creamy bokeh of [色调] shapes without revealing specific details`,禁止写具体道具名+材质+位置(Image_2 豁免) | 全 | M2 Step 9 | 该图重写 |

### 新增/调整条款扩充约定

未来新增硬约束 → 直接在本表追加 H13/H14...,无需修改下游任何文件。
品类专属硬约束 → 在对应子 SKILL(B2/B1/未来 C 系列)的 §零「品类硬约束清单」内声明,使用 H-B/H-A/H-C 等前缀编号。

## 二、标准输入

```
必填:产品图(1 张或多张,建议 3-5 张覆盖正/侧/背/顶/细节)/ 站点 / 类目 / 材质 / 功能属性 / 尺寸
可选:评论分析报告 / 关键词搜索报告 / 竞品差异化 / 转化目标
工作模式:图生图(image-to-image),不可更改
```

数据缺失时,Skill 基于产品图与品类常识自行判断,并在最终输出末尾提示用户:
"未提供数据文件,本次基于品类常识判断,建议补充数据以提高准确度。"

## 八、用户覆盖处理协议

> 当用户的显式要求与本 SKILL 的默认硬约束冲突时,按本协议处理。
> 不得在用户没有显式覆盖的情况下擅自跳过任何硬约束。
> 不得在用户显式覆盖时静默照做不警告。

### 8.1 覆盖类型分级

| 等级 | 定义 | 例子 | 处理 |
|---|---|---|---|
| **L1 软覆盖** | 改变默认参数,不打破任何硬约束 | 改景别比例 / 改空间数量 / 指定具体职责数量 / 改风格倾向 | 直接执行,不警告 |
| **L2 硬约束覆盖** | 打破主 SKILL 总原则第 9-11 条或子 SKILL 的硬约束 | 取消 Image_2 尺寸图 / 取消交互图 / 全部走场景图无置景 / 花瓶全部空瓶 / 容器全部装物 | 执行前列出被打破的硬约束 + 一句话商业后果 + 输出末尾标注「非标准主图副图集」 |
| **L3 元规则覆盖** | 试图打破第一铁律 / 视觉哲学 / C1 协议本身 | 让产品本体描述写颜色纹理 / 跳过审核 / 自动改 SKILL 文件 | 拒绝执行,告知用户该规则不可豁免 |

### 8.2 L2 覆盖的执行流程

```
Step 1:识别覆盖意图,显式列出被打破的硬约束
Step 2:一句话说明该硬约束的商业目的(为什么它本来在那里)
Step 3:执行用户要求,不拖延
Step 4:输出末尾追加「非标准主图副图集」标注 + 用途建议(A+ 内容/详情页氛围图等)
```

### 8.3 禁止事项

- 禁止默认用户的覆盖请求是 L1
- 禁止把 L2 当 L1 静默执行
- 禁止 L3 被措辞包装就放行
- 禁止把"用户没说要 X"当成"用户允许跳过 X"

## 九、最终提交方式

全部阶段完成后,按以下两步提交(解决中文编码问题):

1. 用 `write` 工具将完整 JSON 写入临时文件(路径从任务指令获取)
   文件内容格式:`{"task_id":"xxx","data":{...全部阶段结果...}}`
   data 应包含:
   - 阶段1:产品定位包(产品类型 A/B/C/D + 尺寸档位 + 归属空间 + 数据解析结果 + 决策候选)
   - 阶段2:风格包(风格名称 + 视觉参数 + 心智方案 + 职责分配) + 9 张 Image_N 描述 + 整体说明
   - 数据缺失时:在 data 内追加 `"data_missing_notice": "未提供数据文件,本次基于品类常识判断,建议补充数据以提高准确度。"`
2. 用 `exec` 工具执行:`curl -s -X POST <提交URL> -H "Content-Type: application/json" -d @<临时文件路径>`

提交成功返回 `{"ok":true}`,确认成功后再结束任务。
❌ 禁止直接输出文本结果,必须通过 write + curl 提交。
